Webhooks allow real-time integration between the Zynx API and WhatsApp, enabling automated data synchronization and sharing.

This feature makes it possible to create self-service bots and multi-service systems with real-time events.

## Configuring Webhooks per Instance

Each instance can be configured with its own webhooks independently. This allows you to control exactly which events you want to receive for each instance.

### Set Webhook

To configure a webhook on a specific instance, use the `/webhook/set` endpoint:

```json
POST /webhook/set/{instance}

{
  "url": "https://your-domain.com/webhook",
  "webhook_by_events": false,
  "webhook_base64": false,
  "events": [
    "QRCODE_UPDATED",
    "MESSAGES_UPSERT",
    "MESSAGES_UPDATE",
    "MESSAGES_DELETE",
    "SEND_MESSAGE",
    "CONNECTION_UPDATE"
  ]
}
```

### Parameters

| Parameter         | Type    | Required | Description |
|-------------------|---------|----------|-------------|
| url               | string  | Yes      | URL to receive webhook events |
| webhook_by_events | boolean | No       | Generate a specific URL for each event |
| webhook_base64    | boolean | No       | Encode the payload in base64 |
| events            | array   | No       | List of events to monitor |

### Get Webhook Configuration

To check the current webhook configuration for an instance:

```
GET /webhook/find/{instance}
```

Response:

```json
{
  "enabled": true,
  "url": "https://your-domain.com/webhook",
  "webhookByEvents": false,
  "events": [
    "QRCODE_UPDATED",
    "MESSAGES_UPSERT",
    "MESSAGES_UPDATE",
    "MESSAGES_DELETE",
    "SEND_MESSAGE",
    "CONNECTION_UPDATE"
  ]
}
```

## Supported Events

These are all available webhook events:

| Event | Description |
|-------|-------------|
| APPLICATION_STARTUP | Notifies when the application starts |
| QRCODE_UPDATED | Sends the QR code in base64 format |
| CONNECTION_UPDATE | Informs the WhatsApp connection status |
| MESSAGES_SET | Initial list of loaded messages |
| MESSAGES_UPSERT | New message received |
| MESSAGES_UPDATE | Message was updated |
| MESSAGES_DELETE | Message was deleted |
| SEND_MESSAGE | Message was sent |
| CONTACTS_SET | Initial list of contacts |
| CONTACTS_UPSERT | Contacts were reloaded |
| CONTACTS_UPDATE | Contact was updated |
| PRESENCE_UPDATE | User presence status (online, typing, etc) |
| CHATS_SET | Initial list of chats |
| CHATS_UPDATE | Chat was updated |
| CHATS_UPSERT | New chat added |
| CHATS_DELETE | Chat was deleted |
| GROUPS_UPSERT | New group created |
| GROUPS_UPDATE | Group was updated |
| GROUP_PARTICIPANTS_UPDATE | Participant added, removed, or promoted |
| NEW_TOKEN | JWT token was renewed |

## Webhook by Events

When you enable `webhook_by_events`, Zynx automatically appends the event name to the end of your URL.

### Example

If your URL is `https://your-domain.com/webhook/`, with `webhook_by_events` enabled:

- `QRCODE_UPDATED` → `https://your-domain.com/webhook/qrcode-updated`
- `MESSAGES_UPSERT` → `https://your-domain.com/webhook/messages-upsert`
- `CONNECTION_UPDATE` → `https://your-domain.com/webhook/connection-update`

This allows you to have different endpoints for each event type on the same base URL.

## Tips

- Configure different webhooks for each instance based on your needs
- Use `webhook_by_events` if you need different logic for each event type
- Always ensure your webhook URL is accessible and responds with status 200
- Implement retry logic in your application to ensure events are processed
